# 인원요청서 자동 생성 에이전트 설계서

> 영업팀 슬랙 워크플로우 "상담하기"의 자동 게시 메시지를 입력으로 받아, 표준 인원요청 양식을 생성하고 운영 스프레드시트(또는 .xlsx)에 새 시트로 등록하는 에이전트 시스템의 통합 설계서. Claude Code 구현 시 참조하는 계획서.

---

## 1. 작업 컨텍스트

### 1.1 배경
이벤트·축제·BTL·팝업스토어 등 오프라인 행사에 필요한 인력을 공급하는 업무를 수행한다. 현재 흐름은 다음과 같다.

```
영업팀                                  섭외팀
  │                                      │
  │ 슬랙 워크플로우 "상담하기" 실행      │
  ▼                                      │
[채널 자동 생성 + 정형 메시지 게시]      │
  │                                      │
  └─────────── 채널 공유 ───────────────▶│
                                         │ (사람이 메시지 읽고
                                         │  운영 스프레드시트에 옮김)
                                         ▼
                                  [운영 스프레드시트]
```

섭외팀의 "메시지 읽고 시트에 옮기는 작업"이 본 에이전트가 자동화할 영역이다.

### 1.2 목적
- 슬랙 메시지 → 표준 양식 변환 작업의 수작업 제거
- 양식·시트 구조 일관성 확보
- 시트 생성 및 데이터 입력 자동화
- 부가 일정(세팅/사전교육)의 누락 방지

### 1.3 범위
**포함 (v1 MVP)**
- 슬랙 메시지 텍스트 입력 수용 (복사 붙여넣기)
- 메시지 본문에서 표준 필드 + 부가 일정 추출
- 표준 인원요청서 텍스트 생성
- 로컬 `.xlsx` 파일 생성
- 정보 누락/모호 시 섭외팀에 확인 질의 (배치 처리)
- **수정/변경 메시지 처리** — 기존 행사의 인원·일정 변경 (자주 발생)

**포함 (v2 — 이후 확장)**
- 슬랙 링크 입력 수용 (slack-message-fetcher 스킬)
- Google Spreadsheet 새 시트 생성 (gsheet-writer 스킬)

**제외 (전 버전 공통)**
- 인력 배정(이름 채우기) — 사람이 후처리
- 거래처/행사 마스터 DB 관리
- 영업팀 슬랙 워크플로우 자체의 수정·확장
- 이메일 발송 등 후속 액션

### 1.4 입력

**출처**: 영업팀 슬랙 워크플로우 "상담하기"가 채널에 자동 게시하는 정형 메시지

**입력 방식**:
- **(A) v1 MVP**: 섭외팀이 슬랙 메시지 텍스트를 복사해 에이전트에 붙여넣기
- **(B) v2**: 섭외팀이 슬랙 메시지 링크(또는 채널/메시지 식별자)를 제공 → 에이전트가 슬랙에서 직접 조회

**수정 메시지 인식**:
- 메시지 본문에 `수정` 또는 `변경` 키워드가 포함된 경우 수정 케이스로 분류
- 수정 케이스 처리: 기존 `.xlsx` 파일 탐색 → 기존 시트 탭명에 `_수정전` 추가 → 신규 시트 생성
- 기존 파일 탐색 기준: `{거래처명}_{행사명}*.xlsx` 패턴 (output/ 디렉터리 내)
- 매칭 파일 없으면 수정 전 파일 부재 안내 후 신규 케이스로 처리

**메시지에 포함되는 필드**:

| 필드 | 비고 |
|------|------|
| 상담자 (영업팀 담당자) | 슬랙 멘션 형식 |
| 거래처명 | 신규 표시(`*신규`) 포함 가능 |
| 행사명 | |
| 근무 시작일 / 근무 종료일 | YYYY-MM-DD |
| 근무시간 / 인원 / 성별 | 자유 텍스트. 날짜별·인력구분별로 복수 블록 가능 |
| 장소(실내/외) | 상세 주소 포함 |
| 업무 내용 | 자유 텍스트 |
| 근무복장 | "미정" 가능 |
| 복장 제공 여부 | |
| 식사 제공 여부 | |
| 고객 요청사항 | 자유 텍스트. 부가 일정 단서가 있을 수 있음 |
| 상담 특이사항 | 자유 텍스트. 부가 일정·특수 조건 단서가 있을 수 있음 |
| 문의경로 / 유입경로 / 링크 발송 | 옵션 |
| 행사 담당자 이름·직급·연락처·이메일 | 옵션 |

**슬랙 워크플로우 메시지 구체 포맷** (샘플 기반 확정):
```
거래처명: {값} | 행사명: {값}
근무 시작일: YYYY-MM-DD | 근무 종료일: YYYY-MM-DD
근무시간 / 인원 / 성별:
{자유 텍스트 — 번호 목록 또는 자유 서술}
장소(실내/외): {값}
ㅡㅡㅡㅡ (구분선: ㅡ 문자 30개 이상)
업무 내용:
{값}
근무복장:
{값}
복장 제공 여부: {값}
식사 제공 여부: {값}
ㅡㅡㅡㅡ
고객 요청사항:
{자유 텍스트}
상담 특이사항:
{자유 텍스트}
ㅡㅡㅡㅡ
문의경로: {값}
유입경로: {값}
링크 발송: {값}
ㅡㅡㅡㅡ
행사 담당자 이름: {값} | 행사 담당자 직급: {값}
행사 담당자 연락처: {값} | 행사 담당자 이메일: {값}
```

**파싱 규칙**:
- 필드 구분선: `ㅡ` 문자가 10개 이상 연속된 줄
- 동일 행 복수 필드: `|` 문자로 구분 (`필드명: 값 | 필드명: 값`)
- 필드값이 다음 줄에 오는 경우: `필드명:\n{값}` (값이 비어있으면 빈 문자열)
- "근무시간/인원/성별" 블록: 구분선 이전까지의 자유 텍스트 전체

**⚠️ 중요 — 초기 메시지의 스케줄 상세 수준**:
- 초기 워크플로우 메시지는 **총괄 인원만 포함**하는 경우가 많음 (예: "팀장 1인, 영어스탭 8인")
- 날짜별 상세 스케줄은 이후 슬랙 대화에서 별도 전달되는 게 일반적
- 에이전트는 두 가지 상황 모두 처리 가능해야 함:
  - (A) 초기 메시지에 상세 스케줄 포함 → 그대로 추출
  - (B) 초기 메시지에 총괄 인원만 → 에스컬레이션으로 날짜별 상세 요청

**수정 메시지 키워드** (초기 목록보다 확장):
- `수정사항`, `수정`, `변경`, `변동`, `변동사항`
- 키워드가 메시지 앞부분(첫 5줄 이내)에 등장하면 수정 케이스로 분류

**메시지에 명시 필드가 없는 정보** (본문에서 추출 또는 사용자 확인 필요):
- 세팅일 / 사전교육 일정 등 행사 본 일정 이외의 부가 일정
- 날짜별 상세 스케줄 (초기 메시지에 없는 경우)

### 1.5 출력
- **표준 양식 텍스트**: `/output/02_requirement.txt`
  - 구조: 상단 기본정보 블록 + 하단 날짜별 스케줄 표
  - 기존 수작업 양식 기반, 에이전트 출력이 곧 표준 (함께 다듬어 확정)
  - ⚠️ TBD: 양식 샘플 수령 후 세부 형식 확정
- **시트 출력** (v1: xlsx만, v2에서 Sheets 추가):
  - v1: 로컬 `.xlsx` 파일 (`/output/{거래처명}_{행사명}.xlsx`)
  - v2: Google Spreadsheet 기존 파일에 새 시트 추가

### 1.6 제약조건
- 메시지 기본 구조는 정형이나, 자유 텍스트 필드(업무 내용·고객 요청사항·상담 특이사항)에 많은 양의 내용이 들어올 수 있음 → LLM 추출 필요
- **인력구분이 자유 텍스트**: 팀장·스탭·모델·프로모터·의전·경호 등 행사에 따라 다양. 미리 정의된 enum이 아님
- 인원수·날짜 수·인력구분 수가 행사마다 달라 **시트 구조는 동적**이어야 함 (행: 인력구분별 인원, 열: 부가일정 + 행사일)
- 인력 이름 셀은 **빈 상태로 출력** (섭외팀이 후처리)
- 부가 일정(세팅/사전교육 등)이 메시지 본문에 있을 수도, 없을 수도 있음. 추출 실패 시 자동 추정 금지 → 사용자 확인
- **비용/단가는 시트에 없음** — 재경팀이 별도 파일로 관리. 에이전트가 생성하는 시트에 단가·비용 열 불필요
- 시트 레이아웃: 스크린샷 샘플 기반 구조 확정 (§6a 참조). 세부 스타일링은 추후 다듬기 가능
- v2에서만 해당: Google Sheets 사용 시 Service Account 인증 권장, 슬랙 링크 입력 시 Slack API/MCP 권한 필요

### 1.7 용어 정의
- **상담 메시지**: 슬랙 워크플로우 "상담하기"가 생성하는 정형 메시지 (본 에이전트의 입력 원본)
- **인원요청서**: 행사 일정과 필요 인력을 정리한 표준 양식 (본 에이전트의 1차 산출물)
- **인력구분**: 행사에서 수행할 역할 분류. 예: 팀장, 스탭, 모델, 메이크업모델, 프로모터, 의전, 경호 등. 행사마다 달라짐
- **부가 일정**: 행사 본 일정 외에 인력이 투입되는 일정. 예: 세팅일, 사전교육일, 리허설일 등
- **TBD**: To Be Determined. 인력 이름이 미확정 상태인 셀

## 2. 워크플로우 정의

### 2.1 전체 흐름

```
[입력] 슬랙 메시지 텍스트 (v1) 또는 슬랙 링크 (v2)
   │
   ▼
[0] 입력 어댑터 (입력 형태 정규화 + 수정 여부 감지)
   │   ├─ /output/00_raw_message.txt
   │   └─ 수정/변경 키워드 감지
   │         ├─ 수정 케이스 → 기존 xlsx 탐색
   │         │     ├─ 파일 있음 → 기존 시트 탭명에 "_수정전" 추가 후 계속
   │         │     └─ 파일 없음 → 사용자 안내 후 신규로 처리
   │         └─ 신규 케이스 → 그대로 계속
   ▼
[1] 메시지 파싱 및 필드 추출 (LLM)
   │   ├─ 표준 필드 추출
   │   └─ 본문에서 부가 일정 단서 탐색
   │   └─ /output/01_extracted.json
   ▼
[2] 필수 필드 검증 + 부가 일정 확인 (스크립트 + LLM)
   │   ├─ 통과 → 다음
   │   └─ 누락/모호 → [E] 배치 질의 (모든 질문 한 번에) → 재진입
   ▼
[3] 표준 양식 텍스트 생성 (LLM)
   │   └─ /output/02_requirement.txt
   ▼
[4] 양식-JSON 일치 검증 (LLM 자기검증)
   │   └─ 실패 → 자동 재시도 → 실패 시 에스컬레이션
   ▼
[5] 출력 채널 결정 (메인 에이전트 판단)
   │   ┌──────────────────┬──────────────────┐
   ▼                      ▼
[6a] .xlsx 파일 생성 (v1)  [6b] Google Sheets 시트 생성 (v2)
   │                      │
   └──────────────────────┘
   ▼
[7] 결과 보고 (파일 경로/URL + 후처리 안내)
```

**재개(Resume) 동작**: `/output/` 디렉터리의 중간 산출물 파일이 존재하면 해당 단계를 건너뛰고 이어서 시작.
- `00_raw_message.txt` 존재 → [0] 건너뜀
- `01_extracted.json` 존재 → [0], [1] 건너뜀
- `02_requirement.txt` 존재 → [0]~[3] 건너뜀

### 2.2 단계별 정의

#### [0] 입력 어댑터
- **처리 주체**: 메인 에이전트 (LLM 판단) + 스크립트
- **작업**:
  - 입력이 **텍스트 붙여넣기 (v1)**: 그대로 정규화 (구분선 통일, 줄바꿈 정리)
  - 입력이 **슬랙 링크 (v2)**: 슬랙 API/MCP로 해당 메시지 조회 → 본문 텍스트 추출 → 정규화
  - **수정 여부 감지** (정규식): 정규화 텍스트에 `수정|변경` 패턴 포함 시 수정 케이스로 분류
    - 수정 케이스: `/output/{거래처}_{행사}*.xlsx` 탐색 → 파일 존재 시 기존 시트 탭명에 `_수정전` 추가 → 계속 진행
    - 파일 미존재: 사용자에게 "기존 파일을 찾지 못했습니다. 신규로 처리합니다" 안내
- **출력**: `/output/00_raw_message.txt`
- **성공 기준**: 정규화된 텍스트에 식별 가능한 구분선과 필드 레이블이 존재
- **검증 방법**: 규칙 기반 (구분선 패턴 매칭, 핵심 레이블 존재 여부)
- **실패 시 처리**: 텍스트 형식이 양식과 다름 → 사용자에게 형식 안내 후 재요청 / (v2) 슬랙 조회 실패 → 인증/권한 오류 안내 후 에스컬레이션

#### [1] 메시지 파싱 및 필드 추출
- **처리 주체**: LLM
- **작업**:
  - 정규화 텍스트에서 표준 필드를 JSON으로 추출
  - 본문(업무 내용·고객 요청사항·상담 특이사항)에서 **부가 일정 단서** 탐색 (예: "사전교육 5/27", "전일 세팅 필요" 등)
  - "근무시간/인원/성별" 자유 텍스트 블록에서 (날짜, 인력구분, 인원수, 시작시간, 종료시간, 성별) 튜플 리스트로 정규화
- **출력**: `/output/01_extracted.json`
- **성공 기준**: 모든 필수 키 존재. 인력 튜플 리스트가 최소 1개 이상
- **검증 방법**: 스키마 검증 (필수 키, 타입, 날짜·시간 형식)
- **실패 시 처리**:
  - 형식 오류 → **자동 재시도 1회**
  - 재시도 후에도 모호 → **에스컬레이션**

#### [2] 필수 필드 검증 + 부가 일정 확인
- **처리 주체**: 스크립트 (스키마·규칙) + LLM (의미·부가 일정 판단)
- **작업**:
  - 규칙 검증: 날짜 형식, 시간 범위, 인원수 양수, 종료일 ≥ 시작일
  - 의미 검증: 인력구분 표기 일관성, 시간대 충돌 없음
  - 부가 일정 판단: [1]에서 추출된 단서가 있으면 사용자에게 "이 단서가 세팅/사전교육 일정으로 보이는데 맞는지" 확인. 단서가 전혀 없으면 명시적으로 "부가 일정 없음"으로 처리 vs 사용자 확인 — 정책은 §2.4 참조
- **성공 기준**: 모든 필수 필드 통과 + 부가 일정 상태 명확화(있음/없음/추가됨)
- **검증 방법**: 규칙 기반 + LLM 자기검증
- **실패 시 처리**: **배치 에스컬레이션** — 모든 모호 항목을 번호 목록으로 한 번에 사용자에게 제시. 개별 항목 질의 금지

#### [3] 표준 양식 텍스트 생성
- **처리 주체**: LLM
- **작업**: JSON → 합의된 인원요청서 양식 텍스트 변환. 부가 일정이 있으면 양식에 별도 섹션으로 포함
- **출력**: `/output/02_requirement.txt`
- **성공 기준**: 양식의 모든 섹션 헤더 포함, 일관된 형식
- **검증 방법**: 규칙 기반 (필수 섹션 헤더 존재) + LLM 자기검증 (가독성)
- **실패 시 처리**: **자동 재시도 1회**

#### [4] 양식-JSON 일치 검증
- **처리 주체**: LLM 자기검증
- **작업**: 추출 JSON과 생성 양식 텍스트 간 정보 손실/왜곡 여부 확인
- **성공 기준**: JSON의 모든 값이 양식에 반영, 임의 정보 추가 없음
- **검증 방법**: LLM 자기검증
- **실패 시 처리**: **자동 재시도 1회** → 실패 시 에스컬레이션

#### [5] 출력 채널 결정
- **처리 주체**: 메인 에이전트 (LLM 판단)
- **판단 규칙**:
  - **v1 MVP**: 항상 `.xlsx` 로컬 파일 생성
  - **(v2) 사용자 명시 채널 있음** → 그대로
  - **(v2) 명시 없음 + Google Sheets 인증 정보 존재** → Sheets 기본
  - **(v2) 인증 정보 없음** → `.xlsx` fallback
  - **(v2) "둘 다" 요청** → 병행 실행

#### [6a] .xlsx 파일 생성 (v1 MVP)
- **처리 주체**: 스크립트 (`xlsx-writer`)
- **시트 레이아웃 구조** (스크린샷 기반 확정):
  ```
  행 1: 날짜 헤더 (17일, 18일, 19일, ...)
  행 2: 요일 헤더 (일, 월, 화, ...)
  행 3~: 특수 행 (세팅, 오픈, 포행사, 철수 등 — 부가 일정)
  이후 행: 인력 구성
  ├─ 왼쪽 열: [그룹명] | [역할명] | [이름] (빈 셀)
  ├─ 날짜 열: 근무시간 ("HH:MM - HH:MM" 또는 "HH:MM - 익일 HH:MM")
  └─ 오른쪽 열: 연락처 (빈 셀)
  ```
- **작업**: openpyxl로 워크북 생성 → 시트 구조 그리기 → 데이터 입력
  - 날짜/요일 헤더 행 생성
  - 좌측 정보 블록 작성 (거래처·행사명·근무일·세팅일·담당자·연락처·이메일·장소)
  - 인력구분별로 인원수만큼 행 동적 생성 (예: 팀장1, 팀장2, 스탭1~5, 모델1)
  - 각 날짜 셀에 근무시간 입력
  - 이름 열·연락처 열: 빈 셀 (섭외팀이 나중에 채움)
- **비용 열 미포함**: 단가/비용은 재경팀이 별도 파일로 관리 — 에이전트 시트에 불필요
- **시트 이름 규칙**: `{거래처명}_{행사명}` (충돌 시 `_{YYYYMMDD}`)
- **출력**: `/output/{거래처명}_{행사명}.xlsx`
- **성공 기준**: 파일 생성, 재로드 시 셀 값 일치
- **검증 방법**: 규칙 기반 (파일 존재 + 재로드 검증)
- **실패 시 처리**: **자동 재시도 1회**

#### [6b] Google Sheets 시트 생성 (v2)
- **처리 주체**: 스크립트 (`gsheet-writer`)
- **작업**: [6a]와 동일한 구조를 Google Sheets API로 생성
- **인증 방식**: Service Account (JSON 키 파일) 권장
- **성공 기준**: API 응답 정상, 시트 URL 반환, 재조회 시 셀 값 일치
- **검증 방법**: 규칙 기반 (응답 코드 + 재조회 검증)
- **실패 시 처리**: 인증 오류 → 에스컬레이션 / 그 외 → **자동 재시도 2회**

#### [7] 결과 보고
- **처리 주체**: 메인 에이전트
- **작업**: 시트 URL/파일 경로 + 후처리 안내(인력 이름 채우기, 부가 일정 반영 여부 확인) 사용자에게 전달

### 2.3 LLM 판단 vs 코드 처리 구분

| 단계 | 처리 방식 |
|------|----------|
| 0. 입력 형태 판별 | **LLM** (텍스트 vs 링크) |
| 0. 수정/변경 키워드 감지 | 스크립트 (정규식) |
| 0. 기존 xlsx 파일 탐색 | 스크립트 (파일시스템) |
| 0. 슬랙 메시지 조회 (v2) | 스크립트 (Slack API/MCP) |
| 0. 텍스트 정규화 | 스크립트 |
| 1. 필드 추출 + 부가 일정 단서 탐색 | **LLM** |
| 1. 스키마 검증 | 스크립트 |
| 2. 규칙 검증 | 스크립트 |
| 2. 의미·부가 일정 판단 | **LLM** |
| 2. 배치 에스컬레이션 질의 | **LLM** (질의 항목 취합·정리) |
| 3. 양식 텍스트 생성 | **LLM** |
| 4. 양식-JSON 일치 검증 | **LLM 자기검증** |
| 5. 출력 채널 결정 | **LLM** |
| 6a. xlsx 파일 생성 (v1) | 스크립트 (openpyxl) |
| 6b. Sheets 시트 생성 (v2) | 스크립트 (Google Sheets API) |
| 7. 결과 정리 보고 | **LLM** |

### 2.4 에스컬레이션 트리거

**질의 방식**: 에스컬레이션 발생 시 모든 확인 항목을 **한 번에 번호 목록으로 취합하여** 사용자에게 제시. 항목별 개별 질의 금지.

예시 형식:
```
다음 항목들을 확인해 주세요:
1. [필드명] ○○이 불명확합니다. → ?
2. [부가일정] 특이사항에 "전일 세팅" 언급이 있습니다. 세팅일 시간/장소를 알려주세요.
3. [인력구분] "VIP 안내"는 처음 보는 표기입니다. 그대로 사용할까요?
```

| 상황 | 사용자에게 묻는 내용 |
|------|----------------------|
| 필수 필드 누락 (재시도 후에도) | 어떤 필드가 비어있고, 어떻게 채울지 |
| 날짜·시간 표기 모호 | 예: "5/28 13:00-13:40"이 단일 슬롯인지 등 |
| 본문에 부가 일정 단서가 있음 | "사전교육/세팅 일정으로 보이는데 맞는지, 시간/장소는?" |
| 본문에 부가 일정 단서가 없음 (정책상) | "세팅이나 사전교육 일정이 필요한 행사인지" 한 번 확인 |
| 인력구분이 기존에 본 적 없는 신규 표기 | 그대로 사용할지, 표준 명칭으로 변환할지 |
| 수정 케이스인데 기존 파일 없음 | "기존 파일을 찾지 못했습니다. 신규로 처리하겠습니다" 안내 (확인 불필요) |
| 출력 채널 인증 실패 (v2) | xlsx fallback할지, 인증 재설정할지 |
| 슬랙 메시지 조회 실패 (v2) | 권한·링크 확인 요청 |

### 3.2 에이전트 구조

**단일 에이전트** (서브에이전트 미사용)

- 스킬 분리만으로 컨텍스트 효율 충분

- 에이전트 실행 시 `/output/` 디렉터리의 중간 산출물 파일 존재 여부 먼저 확인
- 파일 존재 시 해당 단계를 건너뛰고 다음 단계부터 이어서 시작
  - `00_raw_message.txt` 있음 → [0] 스킵

  - `02_requirement.txt` 있음 → [0]~[3] 스킵
- 사용자가 처음부터 다시 실행하길 원하면 `/output/` 파일 삭제 후 재실행

**인터페이스 (v1)**:
- Claude Code CLI — 섭외팀 직원이 터미널에서 직접 실행
- 추후 슬랙 봇 래핑 고려 (v2 이후)

**코드/문서 언어 규칙**:
- 코드 (변수명, 함수명, 코드 코멘트): 영어

### 3.5 데이터 전달 방식

**파일 기반** 원칙. 각 단계의 산출물은 `/output/` 하위에 저장, 다음 단계는 파일 경로 참조.

- 인라인 전달은 단계 간 상태 신호(입력 형태, 채널 선택값 등) 정도로 제한
- 중간 산출물 파일은 디버깅·재실행에도 활용

### 3.6 주요 산출물 파일 형식

| 파일 | 형식 | 내용 |
|------|------|------|

## 4. 합리적 기본값 (사용자 미지정 항목)

| 항목 | 기본값 | 근거 |
|------|--------|------|
| 시트 이름 규칙 | `{거래처명}_{행사명}` (충돌 시 `_{YYYYMMDD}`) | 식별성 + 간결성 |
| 인력 이름 표기 | 모든 셀 빈 상태 | "사람이 후처리" 원칙 |
| 시간대 표기 | `HH:MM - HH:MM` | 사용자 제공 시트 예시와 일치 |
| 필드 누락 처리 | 1회 재시도 → 배치 에스컬레이션 | "정보 부족 시 사람 확인" 원칙 |
| 에스컬레이션 질의 방식 | 모든 확인 항목을 번호 목록으로 한 번에 | 섭외팀 작업 흐름 최소 방해 |
| 부가 일정 단서 없을 때 | 사용자에게 1회 확인 후 "없음"으로 처리 | 누락 위험 방지 |
| 출력 채널 | **v1: xlsx만** / v2: Sheets > xlsx (인증 없으면 fallback) | v1 MVP 단순화 |
| 수정 케이스 처리 | 새 시트 생성 + 기존 탭명에 `_수정전` 추가 | 실수로 인한 기존 데이터 유실 방지 |
| 시트 가로축 | 부가 일정(있을 시) + 행사 기간 모든 날짜 | 사용자 제공 시트 예시 구조 |
| 시트 세로축 | 인력구분별로 인원수만큼 행 동적 생성 | 인력구분 자유 텍스트 대응 |
| 인력구분 표기 | 슬랙 메시지 원문 표기 보존 | 신규 구분이 자주 등장하므로 표준화 강제는 부적절 |
| 시급/단가 | 시트에 포함 안 함 | 별도 관리 원칙 |
